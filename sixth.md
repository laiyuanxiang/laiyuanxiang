# 第六章：数据与作用域模型

Mindloom 的执行系统遵循 **节点级数据隔离模型（Node-Level Data Isolation Model）**。  
所有数据仅存在于 **Node 的局部作用域**，并通过 **CALL 的输入输出映射**进行显式传递。

在该模型中：

- **Node 是唯一的数据作用域单位**
- **节点之间不存在共享状态**
- **数据只能通过 CALL 进行传递**
- **数据生命周期与 Node 生命周期一致**

这一设计确保了执行系统的数据流动具有明确边界，使整个执行结构具备良好的可追踪性与可推理性。

---

## 6.1 数据模型概述

Mindloom 的数据模型建立在 **强隔离与显式传递原则**之上。

系统中的每一个执行节点在运行时都拥有独立的数据空间。节点之间不会共享数据，也无法直接访问其他节点的内部变量。所有数据交换必须通过 **CALL** 的输入输出映射进行。

这一模型具有以下特征：

| 特征 | 说明 |
|-----|-----|
| 作用域隔离 | 每个 Node 拥有独立的数据空间 |
| 显式数据流 | 数据只能通过 CALL 输入输出传递 |
| 无共享状态 | 节点之间不存在共享变量 |
| 生命周期绑定 | 数据生命周期与 Node 生命周期一致 |

通过这些规则，Mindloom 能够在复杂执行流程中保持清晰的数据依赖关系，并避免共享状态带来的不确定性。

---

## 6.2 Node 数据作用域

在 Mindloom 中，**Node 是唯一的数据作用域单位**。

每个 Node 在运行时都会被分配独立的数据空间，该空间用于存储节点执行过程中使用的所有变量。节点内部的数据统一存储在一个变量域中：

```

local

```

Node 内部的数据通常经历三个阶段：

| 数据阶段 | 描述 |
|---------|------|
| inputs | 调用节点传入的输入参数 |
| local | 节点执行过程中使用与修改的变量 |
| outputs | 节点执行结束时返回的结果 |

当 Node 被创建时：

1. `inputs` 参数被复制到 `local`
2. 节点执行逻辑只允许操作 `local`
3. 节点结束时从 `local` 生成 `outputs`

Node 数据结构示意：

```

inputs → local → outputs

```

在执行过程中：

- Node 可以自由读取或修改 `local`
- 其他 Node 无法访问该节点的任何数据

这种设计使每个执行节点都具备完全独立的数据上下文。

---

## 6.3 CALL 数据传递语义

在 Mindloom 中，**CALL 是唯一的数据传递机制**。

节点之间不能直接访问彼此的数据，所有数据交换都必须通过 CALL 的 **输入映射（input mapping）** 与 **输出映射（output mapping）** 完成。

一个 CALL 定义通常包含以下结构：

```

call:
unit: tool.search
input:
query: keyword
output:
result: search_result

```

该结构定义了两个映射关系：

| 映射类型 | 作用 |
|---------|------|
| input mapping | 父节点数据 → 子节点输入 |
| output mapping | 子节点输出 → 父节点变量 |

数据流示意：

```

Parent Node
│
│ input mapping
▼
Child Node

```

执行流程如下：

1. 父节点从 `local` 读取变量
2. 根据 `input mapping` 生成子节点 `inputs`
3. 子节点执行并生成 `outputs`
4. 根据 `output mapping` 更新父节点 `local`

通过这种机制，所有数据依赖关系都明确体现在执行结构中。

---

## 6.4 参数传递模型

Mindloom 的参数传递采用 **值复制语义（Value Copy Semantics）**。

当一个节点发起 CALL 时，传递给子节点的数据会被复制为新的输入参数，而不是以引用方式共享。

该语义具有以下特点：

| 特性 | 说明 |
|-----|-----|
| 无引用传递 | 子节点无法修改父节点变量 |
| 数据独立副本 | 每个 Node 持有独立数据 |
| 无共享变量 | 不存在跨节点共享状态 |

例如：

```

parent.local.keyword → child.inputs.query

```

在子节点内部：

- `query` 是 `keyword` 的副本
- 修改 `query` 不会影响父节点变量

这种设计保证了节点之间的数据边界清晰，并避免复杂的共享状态问题。

---

## 6.5 数据生命周期

Mindloom 的数据生命周期与 **Node 生命周期**严格绑定。

Node 生命周期可以抽象为以下阶段：

```

Node Created
→ Node Execution
→ Node Finished
→ Node Destroyed

```

对应的数据生命周期：

| 阶段 | 描述 |
|----|----|
| Local Created | Node 创建时初始化数据空间 |
| Runtime Update | 执行过程中更新 `local` |
| Outputs Produced | 执行结束生成 `outputs` |
| Local Destroyed | Node 销毁后数据消失 |

当 Node 执行结束后：

- Node 的数据空间会被销毁
- `local` 中的所有变量失效
- 这些数据不再可被访问

因此，Mindloom 运行时不会产生长期驻留的内部状态。

---

## 6.6 持久状态与外部数据

Mindloom 的执行系统本身 **不提供持久数据存储能力**。

Node 内部的数据只在节点生命周期内存在。如果需要长期保存数据，必须通过外部系统实现。

常见方式包括：

| 数据类型 | 存储位置 |
|---------|---------|
| Node 临时数据 | Node local |
| 持久业务数据 | 数据库 / 文件系统 |
| 外部系统状态 | Action 接口 |

这些外部交互通常通过以下执行器实现：

- **Tool**：访问数据库、文件系统等
- **Action**：调用外部服务或设备

通过这种设计，Mindloom 将运行时执行状态与持久数据存储明确分离。

## 6.7 并行执行的数据语义

在并行流程中，多个 CALL 可以同时执行。

并行执行 **不会改变 Mindloom 的数据隔离原则**。每个并行节点仍然拥有独立的数据作用域，并按照相同的 CALL 机制进行数据传递。

需要注意的是：

当多个 CALL 将结果写入同一父节点变量时，可能产生 **写入冲突**。

规范建议：

- 避免多个 CALL 写入同一变量
- 使用独立变量存储不同结果
- 通过后续流程节点进行结果聚合

通过这种方式，可以保持执行结构的数据依赖关系清晰，并避免并行执行带来的不确定行为。
